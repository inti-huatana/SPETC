\section{The main window}

The window is organised in five columns, left to right.

\subsection{Session}
\label{sec:session}

At the top of the first column the \textbf{SESSION} box names an output
directory. It is pre-filled with \code{spetc\_YYYYMMDDThhmm} from the
current time (editable). Pressing \textsf{Create new session} makes
\code{output/<session name>/} and, from that moment, every run records into
it: the configuration (\code{config.json}), the selected-time result and the
full time series as timestamped CSVs, and the scientific plots as PNG. A
dialog confirms that recording has started. Until a session is created,
nothing is auto-recorded (you can still export manually), and the save
dialogs default to the session directory once one exists.

\subsection{Observation, site and target}
\label{sec:obs-site-target}

Site selector (the built-in observatories, stored in a single
\code{observatories.json} that also carries each site's timezone and UTC
offset, so choosing a site sets the observation time reference
automatically; \textsf{Save site} and \textsf{Import\ldots} manage the
list, and \textsf{Google Earth} opens the browser at the current
coordinates), geographic coordinates and elevation. The
\textbf{observation date} row is compact: \textsf{Date}, \textsf{Time} and
a \code{local}/\code{UT} selector on one line, with the local-time source
(IANA name with DST, e.g.\ \code{Europe/Rome}, or a fixed UTC offset)
below. The target box holds the RA/Dec fields (decimal degrees or
\code{hh:mm:ss} / \code{dd:mm:ss}) with the optional SIMBAD \textsf{Resolve
name} button, the magnitude and reference filter, and the target's
\textbf{radial velocity} and \textbf{E(B$-$V)}. Running the ETC itself
never uses the network: only the coordinates in the fields count.

The OBSERVATORY panel also hosts the \textbf{horizon-profile generator}.
Enter a search radius (1--100, default 10, in km or miles via the unit
selector) and press \textsf{Generate horizon (DEM)} (next to
\textsf{Display horizon} on the same row): the program
downloads the Copernicus GLO-30 elevation model around your coordinates
(cached locally, so only the first run of an area needs the internet),
and traces the apparent terrain horizon --- mountains, hills, the
curvature of the Earth and standard refraction included --- in 360
azimuth steps. The computation runs in the background and reports the
highest obstruction when done; the profile is \emph{automatically} saved
both as a CSV and as a PNG plot in \code{data/horizons/} (format in
Sect.~\ref{sec:file-horizon}). The horizon elevation is constrained to the
physical range $-20^\circ$ (the view down from a very high summit) to
$+90^\circ$. \textsf{Display horizon} plots the saved profile for the
current coordinates at any time. When a saved horizon exists for the
current site it is also drawn on the scientific plots: as the terrain
outline on the sky path, and as a shaded exclusion band on the
altitude-versus-time panel, so the target's visibility is shown against
the \emph{local} horizon and not merely the geometric one. The generator needs two
extra libraries (\code{pip install rasterio pyproj}); everything else in
the program works without them.

\subsection{Telescope, detector and conditions}

The \textbf{Telescope} box holds only the optics: primary diameter, central
obstruction, focal length, scalar optics throughput (excluding QE), and,
on the row below it, an optional calibrated instrument-response curve
(\textsf{Browse} plus a wavelength-unit selector). The scalar throughput
and the calibrated response are \emph{mutually exclusive}: loading a
response curve greys the scalar field (and the engine uses the curve
alone, so the two are never multiplied together), and clearing the curve
restores the scalar. The
separate \textbf{Detector} box holds pixel size, gain, full well, ADC bit
depth, read noise and dark current; the detector dimensions
\textbf{L$\times$W} in pixels (length along dispersion, width across it,
both $>0$, default $1024\times1024$) used to check whether a spectrum and
its diffraction orders fit on the chip; a \textbf{Load QE CSV} button that reads
the detector quantum efficiency from a two-column (wavelength, QE) file (the
status beside it shows the loaded file, or \code{qe.dat} loaded by default);
the QE wavelength unit; the \textbf{sensor-type} row --- a
\code{CCD}/\code{CMOS} technology selector, a compact \code{mono}/\code{color}
selector, and a \textbf{CH:} R/G/B channel selector (greyed out for a mono
sensor) --- where \code{mono} is a monochrome CCD/CMOS and \code{color} is a
one-shot-colour (Bayer) sensor read one channel at a time; aperture rates
and pixel counts scale by the channel's Bayer fill fraction, saturation
stays per physical pixel, and the QE curve should be that channel's
effective QE (sensor QE $\times$ colour-filter dye). A
\textbf{non-linearity limit} field (ADU) marks where the detector's
response stops being linear: leave it empty to default to the ADC ceiling
$2^{\text{bits}}$ (so nothing is flagged), or enter any
$0 < \text{value} \le 2^{\text{bits}}$; counts above it but below hard
saturation are flagged \code{NON\_LIN} in the SAT column, and the outputs
tab reports how many frames are linear, non-linear or saturated (the old,
meaningless \code{BOTH} joint flag is now the explicit \code{FW+ADC}). At
the end of
the Detector box the \textbf{Save profile} / \textbf{Load profile} buttons
bundle the whole telescope + detector + QE + response set into a portable
JSON. A \textbf{CMOS gain table} (a text file with one row per gain
setting: setting, e$^-$/ADU, read noise, full well — published by camera
makers and measured on PhotonsToPhotos): pick the setting and the three
detector fields fill consistently; the gain-table row is active only when
the sensor technology is \code{CMOS}, and greyed out for a \code{CCD}.
Then: seeing FWHM; the sky-background mode
(\code{ing} = physical night/twilight/day model, \code{fixed\_ab} = your
own observed sky surface brightness, \code{sqm} = your \textbf{zenith SQM
reading} in $V$ mag/arcsec$^2$; a single \textbf{sky mag / SQM mag} field
at the top of the box carries the value for both \code{fixed\_ab} and
\code{sqm} modes, with the \textbf{Bortle class} preset just below it to
fill a typical value; band colours and the Moon model are applied on
top); and the \textbf{PSF model} (\code{gaussian} or \code{moffat} with
its $\beta$; Moffat reproduces the wings of real star images and is
recommended); an optional \textbf{seeing-scaling} checkbox that makes the
seeing vary with wavelength and airmass (Kolmogorov turbulence: blue light
more blurred than red, and worse as the target sets --- with it ticked the
value you enter is the zenith $V$ seeing); an optional \textbf{extra
background} in e$^-$/s/pixel, a catch-all for detector glow, stray light or
ghosting that raises the background noise and the peak-pixel prediction;
a \textbf{guiding rms} in arcsec (0 = perfect tracking) that adds
image-motion blur in quadrature to the seeing --- enter your autoguider's
rms error, or an estimate of the unguided drift per exposure; a
\textbf{telluric bands} checkbox that adds a \emph{built-in parametric}
model of the O$_2$/H$_2$O molecular absorption bands of the red and
near-infrared (the deep A band at 7605\,\AA{} in particular) on top of the
smooth extinction curve --- no data file is needed, and it is an
approximation for red/NIR S/N budgeting rather than a measured site
spectrum; and the
\textbf{sky annulus pixels} field: the number of pixels you actually use
to measure the background around the star (0 = assume it perfectly
known). A finite value adds the classical $(1+n_{\rm pix}/n_{\rm sky})$
sky-subtraction noise to every background term --- with a small annulus
this is tens of percent of the background noise, so faint-target
predictions become noticeably more honest.

\subsection{Calculation settings}

Photometry or spectroscopy; exposure time; optional \textbf{target S/N}
(when set, the ETC returns the required exposure instead, flagging
saturation — the solver includes scintillation, so bright-star requests
are honest); the \textbf{stack sub-exposure} preference (0 = automatic)
feeding the stack planner, whose result appears in the label below after
each run. In the photometry panel the \textbf{source geometry} is the
first row and decides which of the fields below it apply: \code{point}
uses the aperture radius (PSF aperture losses); \code{extended} uses the
source area in arcsec$^2$ for galaxies, nebulae and planets (the magnitude
you enter stays the \emph{total} integrated magnitude, spread uniformly
over that area); and \code{defocus} models a \emph{defocused} telescope,
where the star is deliberately spread into a donut to beat scintillation
and flat-field errors. In \code{defocus} mode you enter the
\textbf{focus position} in \textmu m (0 = focused; a positive or negative
value moves the detector intra- or extra-focally) and press
\textsf{Calculate defocused PSF}: a window shows the geometric donut
profile and its encircled-energy curve
(Sect.~\ref{sec:defocus-guide}), from which you pick the photometry
aperture radius; that radius is then used in the run and reported in the
Outputs tab. The three geometries are mutually exclusive in the UI: the
fields that do not apply to the current choice are greyed out (aperture
radius for \code{extended}/\code{defocus}, source area for
\code{point}/\code{defocus}, focus position and the PSF button for
\code{point}/\code{extended}). The panel also has the optional
\textbf{comparison-star magnitude} for differential photometry — the
mmag-per-frame precision appears in the label below. The spectroscopy
panel holds the resolving power, slit width, extraction height, sampling,
the optional \textbf{calibrated slit resolving-power curve} $R(\text{slit
width})$ (moved here from the detector box, since it belongs to the
spectroscopic optics), the slitless parameters
(Sect.~\ref{sec:spec-guide}), the \textbf{WL min/max/ref} on one row with
an \textbf{AA/nm/um} unit selector, the \textbf{slit orientation}
(\code{parallactic} or \code{fixed}), and the grating efficiency. The fields of the mode you are not using are greyed
out: choose \code{slit} and the slitless fields disable, choose
\code{slitless} and the slit fields disable. Likewise, choosing photometry
in the CALCULATION box greys the whole SPECTROSCOPY box, and choosing
spectroscopy greys the whole PHOTOMETRY box. A \emph{single} \textbf{grating
lines/mm} field feeds both the slitless disperser and the Littrow geometry
helper. Grating efficiency is one row --- a \code{value}/\code{file}
selector, the scalar entry, and \textsf{Load CSV\ldots}: with \code{value}
the scalar applies and the load button is greyed; with \code{file} the
loaded wavelength-dependent curve applies and the scalar entry is greyed. The \textbf{spectrograph geometry helper} takes the
collimator/camera focal lengths and that grating lines/mm value; press
\textsf{Compute R from geometry} and the Littrow grating equation fills the
R field (which stays editable) and reports whether you are slit- or
seeing-limited. The optional \textbf{clamp} checkbox goes one step further:
with it ticked, the engine itself limits the entered R to what the geometry
can deliver, so a typo like R $=$ 100\,000 on a 600 l/mm grating cannot
silently inflate every prediction.

\subsection{Filter and template selection}

Two independent filters: the \textbf{reference} filter, in which your
magnitude is expressed (Vega or AB), and the \textbf{observing} filter,
physically in the beam for both photometry and spectroscopy (spectroscopy
defaults to \code{BLANK}, the unfiltered response). Besides the normal
photometric filters (each a \code{\_Vega}/\code{\_AB} profile pair), the
selector also lists single-file \emph{instrumental} passbands that have no
magnitude of their own and are calibrated on the AB scale: \code{BLANK}
(unfiltered), \code{DARK} (opaque) and neutral-density filters
\code{ND25}, \code{ND50}, … Drop their \code{.xml} into the filters
directory and list them in \code{filters.list} to have them appear. The panel previews the
observing transmission curve with its pivot wavelength, width and zero
point; the template selector searches the catalogue by name (showing five
rows at a time) and previews
the selected flux distribution over the fixed \SIrange{3000}{9000}{\angstrom}
optical window (so every template is directly comparable); \textsf{Validate
V and B$-$V} checks the template's synthetic colours against the catalogue.
Immediately below the template box is the \textbf{system-response} preview
(telescope $\times$ optics $\times$ QE $\times$ observing filter), which
now sits in this same column next to the template it modifies rather than
in the instrument column.
The \textbf{radial velocity} and \textbf{E(B$-$V)} that transform the
template are entered in the \emph{target} box (first column,
Sect.~\ref{sec:obs-site-target}), because they are properties of the
target even though they act on
the template: the radial velocity (km/s, positive receding) shifts all
wavelengths by $(1+v/c)$, and E(B$-$V) reddens the template with the
standard CCM89 interstellar extinction law ($R_V = 3.1$). The magnitude
you enter always stays the \emph{observed} one; the transforms only
change the template's colours, exactly as reddening and motion do to a
real star. After a run the outputs tab reports the resulting synthetic
$(B-V)_0$ of the unreddened template and the reddened $(B-V)$, and — for
spectroscopy — the RV shift in pixels and the velocity sampling
(km/s per pixel) at 4000, 5000, 6000 and \SI{7000}{\angstrom}.

\subsection{Run actions and utilities}

The fifth column has \textsf{Run ETC} and \textsf{Exit} (which saves the
configuration) side by side on one row, the live observatory status
(local time, LST, MJD/JD, Sun altitude), and three small converters. The
\textbf{time converter} takes a wide ISO/ISOT/MJD/JD input (defaulting to
the moment the program started) with a \textsf{Now} button, and
\textbf{Input}/\textbf{Output} format selectors on one row. The
\textbf{coordinate converter} turns a DMS latitude/longitude into decimal
degrees. The \textbf{length converter} converts a positive value (negative
input is taken as its absolute value) between any two of inch, foot, yard,
mile, \aa ngstr\"om, nm, \textmu m, mm, cm, m, km, parsec and light-year;
results between $10^{-5}$ and $10^{5}$ are shown in full, otherwise in
exponential notation.
